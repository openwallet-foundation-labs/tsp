## Scope

This set defines a compact body of TSP conformance material that is:

- directly grounded in the TSP specification
- useful for cross-implementation interoperability
- limited to message-layer protocol facts
- structured so that exact and semantic-only comparisons are explicit

This set covers:

- direct, nested, and routed message families
- `byte-exact`, `semantic-only`, and `fixture-only` entries
- complete cases for `HPKE-Auth`, `HPKE-Base`, and `Sealed Box`

The shared vector and fixture catalog is defined independently from case-local artifact freezing. Each complete case declares applicability through its manifest and freezes its own case-local artifacts.

This set does not standardize:

- identity lifecycle policy
- wallet persistence behavior
- transport implementation strategy
- SDK-internal state layout

## Conformance Use

- the specification defines correctness rules
- reviewed case-local vector and fixture assets define authoritative sample material
- compliance tests consume those assets in order to evaluate conformance
- candidate generation does not become authoritative until the resulting material is reviewed and accepted

`CC-001`, `CC-002`, and `CC-003` declare applicability to the shared vector and fixture catalog through case-specific manifests.

## Classification Boundary

The vector schema uses three classes:

- `byte-exact`
  - one authoritative wire artifact is decoded or unwrapped into fixed fields or forwarding material
- `semantic-only`
  - one authoritative wire artifact is opened and compared at the semantic level without requiring identical regenerated bytes
- `fixture-only`
  - identity or conversation material is fixed as supporting reference data for vectors

Mechanism-sensitive review vectors remain within these classes rather than introducing an additional top-level classification.
